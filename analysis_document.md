# 节点式图像处理工作流工具 —— 分析设计文档

## 一、题目理解与设计目标

### 1.1 题目理解
本题要求设计并实现一个**节点式图像处理工作流工具**的桌面原型，用户可以通过可视化方式拖拽连接不同图像处理节点，构建处理流程并一键执行。

核心需求要点：
- **节点式编程范式**：每个处理步骤抽象为一个节点，节点间通过有向连线传递数据
- **图形化界面**：基于 Qt 搭建，支持拖拽、连接、参数编辑、预览
- **工作流持久化**：支持保存/加载工作流（JSON）
- **命令行批处理**：支持无 GUI 模式执行工作流
- **健壮性**：类型检查、错误处理、防崩溃
- **可扩展性**：多态抽象节点，便于后续添加新节点类型

### 1.2 设计目标
- **可用性**：用户能快速搭建工作流并看到结果
- **可扩展性**：添加新节点类型只需继承 Node 基类、注册到工厂
- **稳健性**：类型不匹配无法连接，参数错误有明确提示
- **清晰架构**：GUI、节点逻辑、图像算法、文件读写分离到不同模块

| 类型标识 | 名称 | 内部表示 | 说明 |

| 源\目标 | ColorImage | GrayImage | Any |

交互设计：
- **添加节点**：从节点库面板双击或拖拽节点类型到编辑区
- **移动节点**：在编辑区拖拽节点标题栏
- **连接端口**：从一个输出端口拖拽到另一个输入端口（类型不匹配时拒绝并提示）
- **删除节点/连线**：选中后按 Delete 键或右键菜单
- **参数编辑**：选中节点后，属性面板显示该节点的参数控件
- **预览**：双击节点标题打开预览对话框
- **缩放/平移**：鼠标滚轮缩放，右键拖拽平移

### 5.3 数据流说明

```
1. 用户搭建工作流（图形界面上添加节点、连接端口）
2. 用户点击"执行"按钮
3. WorkflowEngine 从起始节点开始进行拓扑排序
4. 按拓扑顺序依次执行每个节点：
   a. 从连接关系中收集该节点的输入数据
   b. 调用 Node::process(inputs, outputs)
   c. 将输出数据缓存，供下游节点使用
5. 执行完成后将结果显示给用户
```

## 六、异常处理策略

| 异常场景 | 处理方式 |
|----------|----------|
| 文件不存在 | 执行前检查，弹出错误对话框/输出错误信息到控制台 |
| 文件格式不支持 | QImage::load() 失败时返回错误，提示支持的格式 |
| 端口类型不匹配 | 连接时检查数据类型兼容性，不匹配时拒绝连接并提示 |
| 参数越界 | 控件层限制范围（QSpinBox/QSlider 设 min/max） |
| 空输入执行 | 执行前检查所有节点输入是否就绪，缺少时提示 |
| 循环依赖(非DAG) | 拓扑排序检测到环时报告具体环路 |
| 内存不足 | QImage 创建失败时返回空图像，向上传播错误 |
| 命令行参数错误 | 解析参数时输出用法说明 |

整体策略：**尽量让错误在发生点被捕获**，而不是让异常向上传播到顶层。GUI 模式弹出消息框，CLI 模式输出到 stderr 并返回非零退出码。

## 七、AI 工具使用说明

### 7.1 使用的 AI 工具
- 大语言模型辅助编码和设计讨论

### 7.2 AI 对方案的贡献及采纳情况

| 讨论议题 | AI 建议 | 采纳情况 | 理由 |
|----------|---------|----------|------|
| 节点分类设计 | 建议用 FilterNode/ConversionNode 等中间抽象类减少重复代码 | ✅ 采纳 | 确实减少了很多样板代码 |
| 数据类型规范 | 建议加入 Any 通配类型以支持调试节点 | ✅ 采纳 | 提升灵活性 |
| 界面布局 | 建议左侧节点列表 + 中央编辑区 + 右侧属性面板的三栏布局 | ✅ 采纳 | 符合主流节点编辑器设计模式 |
| JSON 格式 | 建议使用 UUID 作为节点标识符 | ✅ 采纳 | 避免重命名冲突 |
| DAG 检测 | 建议用 Kahn 算法进行拓扑排序和环检测 | ✅ 采纳 | 实现简单且高效 |
| 参数控件 | 建议根据参数元数据自动生成编辑控件 | ✅ 采纳 | 减少了每个节点的 UI 代码 |
| 连接线绘制 | 建议用三次贝塞尔曲线 | ✅ 采纳 | 美观且避免遮挡节点 |
| 模块划分 | 建议将图像处理算法抽离为独立工具类 | ✅ 采纳 | 符合"分离到不同模块"的要求 |
| 错误处理 | 建议用返回码而非异常 | ✅ 采纳 | 与 Qt 风格一致，避免 RTTI 开销 |
